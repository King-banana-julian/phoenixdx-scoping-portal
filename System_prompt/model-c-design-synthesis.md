# Model C — Design & Synthesis Activities

## About this model

Model C covers activities where PhoenixDX produces, organises, or visualises design work. These activities are iterative by nature — the number of review and iteration rounds is the primary source of timeline variance, not the initial creation.

**Phase structure:** Brief Review → First Draft → Internal Review → Iteration → Sign-off

**When this model applies:** The team is moving from insights and logic into tangible, reviewable artefacts — maps, diagrams, wireframes, visual designs, or prototypes.

**Critical planning rule:** Never scope iteration as a single round. In Medium projects plan for 2 rounds minimum. In Large and XL projects plan for 3–4 rounds. Iteration time is where most design projects blow out.

**AI acceleration note:** AI is most effective at First Draft stage (generating initial structures, layouts, or component suggestions). Internal Review and Iteration require human designer judgement. Sign-off is always a human activity.

## Card Sorting

**Double Diamond phase:** Define
**Model:** C — Design & Synthesis
**PhoenixDX approach:** Card Sorting is the bridge between a messy content audit and a logical Information Architecture. The synthesis challenge at Large and XL scale is Category Conflict — where Marketing wants a page in one place and Sales wants it in another. User data becomes the referee. Always use a Pilot test for Medium and above.

### Effort by size

| Size | Scope trigger | Effort | AI-accelerated | Friction |
|------|--------------|--------|---------------|---------|
| Small | <30 items, ~10 internal users | 3d | 2d | 1.0x |
| Medium | 50–80 items, 20–30 targeted users | 5d | 3.5d | 1.0x |
| Large | 100+ items, 40–50 users, Hybrid sort | 8d | 5.5d | 1.1x |
| Extra Large | 500+ items, 100+ users, multi-region | 15d | 10.5d | 1.2x |

### Phase breakdown

**Brief Review (Preparation)**
- Small: 1d — Inventory <30 items, tool setup (e.g. OptimalSort), internal pilot
- Medium: 1.5d — Inventory 50–80 items, draft Open vs Closed strategy, recruitment
- Large: 2d — Inventory 100+ items, define complex Hybrid logic, stakeholder alignment
- XL: 4d — Audit 500+ items, multi-language labels, global stakeholder approval

**First Draft (Execution)**
- Small: 0.5d — Send links to ~10 internal/warm users, monitoring
- Medium: 1d — Managed recruitment of 20–30 targeted users, active reminders
- Large: 2.5d — 40–50 users, remote unmoderated plus 5 moderated think-aloud sessions
- XL: 5d — 100+ users across different regions/departments, full session management

**Internal Review (Synthesis)**
- Small: 1d — Standard similarity matrix review, basic menu suggestion
- Medium: 1.5d — Analysis of Dendrograms and Standardisation Grids, pattern matching
- Large: 2.5d — Multi-dimensional scaling (MDS) analysis, resolving cluster conflicts
- XL: 4d — Statistical deep-dive, mapping legacy IA vs New IA, cross-regional variance

**Report & Playback**
- Small: 0.5d — Summary email or short deck with the proposed sitemap
- Medium: 1d — 1-hr workshop showing User Mental Models vs Business logic
- Large: 1d — Formal report with logic justifications and before/after IA visual
- XL: 2d — Executive readout, long-term IA governance strategy and playback

### Deliverables

- **Small:** Summary email or short deck with proposed sitemap based on top user groupings
- **Medium:** 1-hour workshop playback showing User Mental Models vs Business logic with proposed IA
- **Large:** Formal IA report with logic justifications, before/after IA visual, and conflict resolution documentation
- **XL:** Executive readout, long-term IA governance strategy, and cross-regional navigation alignment plan

### Dependencies

- **Small:** Content or feature inventory of fewer than 30 items, tool setup (e.g. OptimalSort), internal participant list
- **Medium:** Inventory of 50–80 items, Open vs Closed sort strategy defined, 20–30 recruited participants
- **Large:** Inventory of 100+ items, Hybrid sort logic defined, 40–50 recruited and segmented participants
- **XL:** Inventory of 500+ items, multi-language labels approved, 100+ recruited participants across regions

### AI validation requirements

- **Small:** Manual validation of AI cluster groupings against actual user intent and mental models
- **Medium:** Dendrogram analysis and pattern matching must be reviewed by a researcher for logical accuracy
- **Large:** Multi-dimensional scaling and cluster conflict resolution require human researcher — AI cannot negotiate department silos
- **XL:** Statistical deep-dive and cross-regional variance analysis must be validated by senior researcher and approved by stakeholders

### Contingency notes

- **Small:** Internal participant recruitment is usually fast but scheduling can still cause delays
- **Medium:** Category conflict between business stakeholders and user data requires facilitation time at playback
- **Large:** Multi-dimensional scaling analysis is time-intensive — underestimating synthesis phase is common
- **XL:** Governance sign-off across competing business units at XL scale requires multiple rounds of diplomacy

## Tree Testing

**Double Diamond phase:** Define
**Model:** C — Design & Synthesis
**PhoenixDX approach:** While Card Sorting helps build the structure, Tree Testing proves it works. We remove visual crutches and ensure the hierarchy can stand on its own. Always run a Pilot Test — if task wording is even slightly leading, data will be skewed and execution must restart.

### Effort by size

| Size | Scope trigger | Effort | AI-accelerated | Friction |
|------|--------------|--------|---------------|---------|
| Small | 2-level tree, 3–5 tasks, ~20 participants | 3d | 2d | 0.9x |
| Medium | 3–4 level tree, 8–10 tasks, 50 users | 5d | 3.5d | 0.9x |
| Large | Complex enterprise tree, 15+ tasks, 100+ users | 10d | 7d | 1.0x |
| Extra Large | Global tree, 20+ tasks, 250+ users, multi-region | 20d | 14d | 1.2x |

### Phase breakdown

**Brief Review (Preparation)**
- Small: 1d — Build a 2-level tree, write 3–5 simple tasks, tool setup
- Medium: 1.5d — 3–4 level tree, 8–10 tasks, pilot testing to refine task wording
- Large: 3d — Complex enterprise tree, 15+ tasks, A/B tree versioning setup
- XL: 6d — Cross-platform/global tree, 20+ tasks, stakeholder alignment on success paths

**First Draft (Execution)**
- Small: 0.5d — Unmoderated, ~20 participants (internal/general)
- Medium: 1d — Unmoderated, 50 users, targeted recruitment via external panel
- Large: 2d — 100+ users, segmented by persona, monitoring for speed-running
- XL: 4d — 250+ users, multiple regions/languages, multi-variant testing

**Internal Review (Synthesis)**
- Small: 1d — Basic success/failure rate review per task
- Medium: 1.5d — Analysis of Pietrees (paths), identifying where users backtracked
- Large: 3d — Correlation analysis (why Persona A fails where Persona B succeeds)
- XL: 6d — Deep statistical analysis, first-click vs final-destination deltas

**Report & Playback**
- Small: 0.5d — Short summary deck, Green/Red light for the sitemap
- Medium: 1d — Playback of Friction Points, recommendations for label changes
- Large: 2d — Detailed IA remediation plan, mapping results back to the Design System
- XL: 4d — Multi-departmental playback, long-term navigation strategy and governance

### Deliverables

- **Small:** Summary deck with Green/Red light assessment and basic success rate per task
- **Medium:** Pietree path analysis, friction point playback, and label change recommendations
- **Large:** Detailed IA remediation plan, correlation analysis by persona, and Design System navigation mapping
- **XL:** Multi-departmental playback, statistical significance report, long-term navigation strategy and governance plan

### Dependencies

- **Small:** Proposed IA or sitemap, 3–5 defined user tasks, tool setup (e.g. Treejack), ~20 participants
- **Medium:** Proposed IA, 8–10 validated tasks after pilot, 50 recruited external participants
- **Large:** Complex enterprise tree, 15+ tasks with defined correct paths, 100+ segmented participants
- **XL:** Cross-platform global tree, 20+ tasks in multiple languages, 250+ participants across regions

### AI validation requirements

- **Small:** Human review of task success rates and basic path analysis
- **Medium:** Pietree path analysis and identification of backtracking patterns require human researcher interpretation
- **Large:** Correlation analysis between persona success rates requires human judgement — AI cannot determine root cause of failure patterns
- **XL:** Deep statistical analysis and first-click vs final-destination deltas require senior researcher validation before recommendations

### Contingency notes

- **Small:** Poorly worded tasks discovered post-launch require restart of execution phase — always pilot first
- **Medium:** Indirect success paths that appear as failures require additional analysis time not always scoped
- **Large:** Duplicate navigation labels in XL trees require complex correct-path logic definition that extends preparation
- **XL:** Multi-region execution and language variants require translation validation and extended recruitment lead times

## Affinity Mapping

**Double Diamond phase:** Define
**Model:** C — Design & Synthesis
**PhoenixDX approach:** Affinity Mapping is the Synthesis Engine. Always colour-code sticky notes by data source (Blue for Interviews, Yellow for Surveys, Pink for Stakeholders). If a cluster is 100% Pink, it is a Business Assumption. If 100% Blue, it is a User Truth. Identifying these imbalances is the most valuable part of the activity.

### Effort by size

| Size | Scope trigger | Effort | AI-accelerated | Friction |
|------|--------------|--------|---------------|---------|
| Small | 3–5 interviews, 3–4 basic themes | 1.5d | 1d | 0.8x |
| Medium | 8–10 interviews + survey, 200+ notes | 4d | 2.5d | 0.8x |
| Large | 20+ interviews, 500+ notes, multi-source | 8d | 5d | 0.9x |
| Extra Large | Global research, 1000+ data points | 17d | 10.5d | 1.0x |

### Phase breakdown

**Brief Review**
- Small: 0.25d — Quick review of notes from 3–5 interviews
- Medium: 0.5d — Auditing notes from 8–10 interviews and 1 survey
- Large: 1d — Reviewing data from 20+ interviews, surveys, and stakeholder sessions
- XL: 2d — Global audit of multi-region research data, aligning on synthesis goals

**First Draft**
- Small: 0.5d — High-level grouping into 3–4 basic themes (e.g. Navigation, Speed)
- Medium: 1.5d — Clustering 200+ notes, identifying I-statements and pain points
- Large: 3.5d — Mapping 500+ notes, multi-layered clustering, cross-persona trends
- XL: 7d — Synthesis of 1000+ data points, mapping global vs regional themes

**Internal Review**
- Small: 0.25d — Peer review to ensure themes aren't just surface level
- Medium: 0.5d — Review with the research team to ensure no outlier insights were lost
- Large: 1.5d — Formal review to validate the Thematic Hierarchy and data integrity
- XL: 3d — Multi-squad review to ensure insights align across different product areas

**Iteration**
- Small: 0.25d — Reframing 1–2 category names for better clarity
- Medium: 0.5d — Refactoring clusters to resolve Thematic Overlap
- Large: 1d — Deep refactoring, moving from Observations to Actionable Insights
- XL: 3d — Large-scale thematic restructuring based on stakeholder feedback loops

**Sign-off**
- Small: 0.25d — Simple walkthrough of Top 3 Insights with the client
- Medium: 1d — Presentation of the Insights Map, securing buy-in for the Problem Space
- Large: 1d — Strategic workshop, using the map to define How Might We statements
- XL: 2d — Executive readout, aligning the roadmap with data-backed user themes

### Deliverables

- **Small:** Digitised affinity map with Top 3 insights and basic theme clusters
- **Medium:** Insights Map with validated theme clusters, I-statements, and How Might We prompts
- **Large:** Strategic affinity map with thematic hierarchy, Insights vs Assumptions balance report, and HMW workshop output
- **XL:** Hierarchical Affinity Map with Macro and Sub-themes, executive summary, and global vs regional insight breakdown

### Dependencies

- **Small:** Notes or transcripts from 3–5 interviews available for clustering
- **Medium:** Research data from 8–10 interviews and at least one survey, collaborative synthesis tool
- **Large:** Raw data from 20+ interviews, surveys, and stakeholder sessions, multi-researcher team available
- **XL:** Global research data from 1000+ data points, hierarchical mapping framework, multi-team coordination

### AI validation requirements

- **Small:** Human must review cluster groupings to ensure themes are genuine insights, not surface-level observations
- **Medium:** I-statement extraction and pain point identification must be validated against source transcripts by researcher
- **Large:** Thematic hierarchy and insight elevation from observations to strategic themes require senior researcher review
- **XL:** Global vs regional theme distinction and macro-theme structuring require full human research team validation

### Contingency notes

- **Small:** Quality of interview notes directly impacts clustering accuracy — poor notes require researcher interpretation
- **Medium:** Thematic overlap between clusters is common and requires reconciliation time
- **Large:** Highly technical domain data requires longer clustering time to avoid misinterpretation of specialist terminology
- **XL:** Stakeholder feedback loops on global themes can cause significant restructuring in late synthesis stages

## Personas

**Double Diamond phase:** Define
**Model:** C — Design & Synthesis
**PhoenixDX approach:** Personas are not marketing profiles — they are Design Targets. Every statement in a Large or XL persona must be traceable back to a transcript. Always include a Scenario for each persona — seeing the persona in action (e.g. Sarah is in a rush between meetings and needs to approve an invoice on her phone with one hand) helps developers and designers visualise constraints far better than a list of hobbies.

### Effort by size

| Size | Scope trigger | Effort | AI-accelerated | Friction |
|------|--------------|--------|---------------|---------|
| Small | 1–2 user types, assumption-based | 1.5d | 1d | 0.9x |
| Medium | 3–4 archetypes, data-backed | 3d | 2d | 0.9x |
| Large | 5–7 complex archetypes, scenario context | 7d | 4.5d | 1.0x |
| Extra Large | 8+ personas, global regions, Archetype Ecosystem | 15d | 10d | 1.1x |

### Phase breakdown

**Brief Review**
- Small: 0.25d — Identify 1–2 primary user types, review existing assumptions
- Medium: 0.5d — Review initial research, identify 3–4 core archetypes
- Large: 1d — Audit interview data, identify 5–7 complex personas across sectors
- XL: 2d — Global stakeholder alignment, review multi-region/multi-product user data

**First Draft**
- Small: 0.5d — Create Proto-Personas (assumption-based), basic bio/goals
- Medium: 1d — Build data-backed profiles, add behaviours and pain points
- Large: 2.5d — High-fidelity personas, add Scenario context and tech-savviness scales
- XL: 5d — Comprehensive Archetype Ecosystem, mapping overlap and distinct triggers

**Internal Review**
- Small: 0.25d — Peer review for clarity and gut check against project goals
- Medium: 0.5d — Review with Sales/Support to check behavioural accuracy
- Large: 1.5d — Cross-functional workshop (Product, Marketing, UX) to validate data
- XL: 3d — Multi-departmental review, ensuring global consistency across regions

**Iteration**
- Small: 0.25d — Minor tweaks to goals or naming
- Medium: 0.5d — Refactoring pain points based on stakeholder feedback
- Large: 1d — Significant shifts to resolve Archetype Overlap (merging or splitting)
- XL: 3d — Large-scale refactoring, managing feedback loops across global teams

**Sign-off**
- Small: 0.25d — Final walkthrough with the primary stakeholder
- Medium: 0.5d — Presentation of the Target Audience, securing alignment for Design
- Large: 1d — Interactive workshop, mapping Personas to specific Product features
- XL: 2d — Executive readout, long-term audience strategy and brand alignment

### Deliverables

- **Small:** Proto-persona documents for 1–2 types with demographics, basic goals, and known pain points
- **Medium:** Research-backed persona documents for 3–4 types with behaviours, pain points, and stakeholder-approved attributes
- **Large:** High-fidelity persona documents with Scenarios, tech-savviness scales, and cross-functional validation workshop output
- **XL:** Comprehensive Archetype Ecosystem document with overlap maps, distinct triggers, global brand alignment, and executive readout

### Dependencies

- **Small:** Basic knowledge of user types, any available research or stakeholder input
- **Medium:** Research synthesis from interviews and surveys, Sales and Support team availability for validation
- **Large:** Completed research synthesis across 5–7 user groups, cross-functional workshop availability
- **XL:** Global research data, multi-region stakeholder input, executive sponsor availability for readout

### AI validation requirements

- **Small:** Human must validate proto-persona attributes against any available research — assumptions must be flagged clearly
- **Medium:** AI-generated persona attributes must be validated against actual interview and survey data by researcher
- **Large:** Scenario context and psychographic accuracy require senior researcher validation against source transcripts
- **XL:** Archetype Ecosystem overlap mapping and global consistency require full cross-functional review and sign-off

### Contingency notes

- **Small:** Stakeholders often challenge proto-personas without data — must be clearly framed as assumption-based
- **Medium:** Stakeholder disagreement on persona validity can stall sign-off — evidence traceability is critical
- **Large:** Archetype overlap between complex personas requires additional synthesis and potential merging or splitting
- **XL:** Global persona alignment across competing regional priorities requires diplomatic facilitation across business units

## Information Architecture

**Double Diamond phase:** Define
**Model:** C — Design & Synthesis
**PhoenixDX approach:** IA is the Skeleton of the product. While UI handles how things look, IA handles how they are grouped and labelled to match the user's mental model. For Large and XL projects, always include a Redirect Strategy — changing an IA in a large enterprise can break thousands of legacy links and destroy SEO.

### Effort by size

| Size | Scope trigger | Effort | AI-accelerated | Friction |
|------|--------------|--------|---------------|---------|
| Small | 10-page product, primary navigation | 2d | 1.5d | 1.0x |
| Medium | 50-page product, 3-level sitemap | 5d | 3.5d | 1.0x |
| Large | 500+ content items, taxonomy planning | 12d | 8d | 1.1x |
| Extra Large | Global multi-site, metadata schema, governance | 25d | 17d | 1.2x |

### Phase breakdown

**Brief Review**
- Small: 0.25d — Review current 10-page site, identify core navigation goals
- Medium: 0.5d — Audit 50+ pages, review Card Sort results if available
- Large: 1d — Deep dive into 500+ content items, reviewing legacy silos
- XL: 2d — Global content audit, reviewing multi-site and multi-language structures

**First Draft**
- Small: 1d — Create a high-level sitemap, define primary labels
- Medium: 2.5d — Develop a 3-level sitemap, define secondary nav and footer
- Large: 6d — Detailed taxonomy, map content types, URL structure planning
- XL: 12d — Comprehensive IA ecosystem, metadata schema, global navigation patterns

**Internal Review**
- Small: 0.25d — Peer review for logical grouping and label clarity
- Medium: 0.5d — SEO and Dev review for structural feasibility and findability
- Large: 1.5d — Review with SEO, Dev, and Content Strategy, check Tree Test data
- XL: 3d — Multi-squad review, ensuring the IA scales across diverse product lines

**Iteration**
- Small: 0.25d — Minor label tweaks or moving 1–2 pages
- Medium: 0.5d — Refactoring categories based on Tree Test failures
- Large: 1.5d — Significant structural shifts to resolve Category Conflict
- XL: 4d — Large-scale taxonomy refactoring based on global stakeholder needs

**Sign-off**
- Small: 0.25d — Simple walkthrough and approval of the sitemap
- Medium: 1d — Formal playback, securing agreement on Top Level navigation
- Large: 2d — High-stakes workshop, justifying labels with user data, migration plan
- XL: 4d — Executive readout, long-term IA governance and maintenance strategy

### Deliverables

- **Small:** High-level sitemap with primary labels and simple walkthrough approval
- **Medium:** Formal 3-level sitemap playback with Top-Level navigation agreement and SEO review notes
- **Large:** Detailed taxonomy document, URL structure plan, content model, IA remediation report with before/after visual
- **XL:** IA ecosystem document, metadata schema, faceted classification model, executive readout, and long-term governance plan

### Dependencies

- **Small:** Basic content list, knowledge of primary navigation goals, stakeholder availability for walkthrough
- **Medium:** Content audit of 50+ pages, Card Sorting results if available, SEO and dev availability for review
- **Large:** Content inventory of 500+ items, Tree Testing results, SEO strategy, content strategy input
- **XL:** Global content audit across all sites, multi-language label validation, regional stakeholder availability

### AI validation requirements

- **Small:** Human review of taxonomy logic and label clarity against user mental models from card sorting data
- **Medium:** SEO and structural feasibility review with dev team must be human-led — AI cannot assess build implications
- **Large:** Significant structural shifts to resolve Category Conflict require human facilitation — AI cannot negotiate department politics
- **XL:** Global IA governance model and redirect strategy require human expert sign-off — AI cannot assess cross-regional content risk

### Contingency notes

- **Small:** Stakeholder disagreement on primary labels can require additional rounds of iteration
- **Medium:** SEO and technical constraints discovered during dev review often require IA refactoring
- **Large:** Category Conflict between business units at scale requires negotiation time using user data as the referee
- **XL:** Content migration redirect strategy planning is commonly forgotten until development and can surface late in the project

## Do/Go Mapping

**Double Diamond phase:** Define
**Model:** C — Design & Synthesis
**PhoenixDX approach:** A Do/Go Map separates Actions (Do) — like Submit, Delete, or Edit — from Navigation (Go) — like Back to Dashboard or Profile Settings. Always define the Undo Strategy early — it is easy to design a Delete button, but the logic of restoring data is complex and if left until build, causes significant rework.

### Effort by size

| Size | Scope trigger | Effort | AI-accelerated | Friction |
|------|--------------|--------|---------------|---------|
| Small | 1–2 core task flows, basic logic | 1.5d | 1d | 0.8x |
| Medium | 5–10 tasks, modal vs page-level actions | 3.5d | 2.5d | 0.8x |
| Large | 20+ complex tasks, error/loading/Undo states | 8.5d | 6d | 1.0x |
| Extra Large | Global enterprise, multi-state logic architecture | 19d | 13d | 1.2x |

### Phase breakdown — see Master Excel Model C - DoGo Mapping tab for full phase details

### Deliverables

- **Small:** Simple Do/Go table covering core tasks with basic success and fail state definitions
- **Medium:** Do/Go matrix with modal vs page-level action mapping, error state definitions, and developer review notes
- **Large:** Comprehensive logic map with error, loading, and Undo state coverage, Stress Test findings, and functional requirements
- **XL:** Multi-state logic architecture document, interaction model sign-off, and executive logic freeze confirmation

### Dependencies

- **Small:** Approved user flows for 1–2 core tasks, basic business requirements
- **Medium:** Approved user flows for 5–10 tasks, API documentation or developer availability for review
- **Large:** Approved user flows for 20+ tasks, Tech Lead availability for Stress Test, regulatory requirements documentation
- **XL:** Full platform architecture documentation, cross-squad engineering availability, global regulatory requirements

### AI validation requirements

- **Small:** Human must validate basic logic for dead-ends and confusing navigation destinations
- **Medium:** AI-generated Do/Go logic must be reviewed by developers to confirm API capability alignment
- **Large:** Technical feasibility Stress Test with Tech Leads is mandatory — AI cannot identify logic races or conflicts
- **XL:** Multi-state logic architecture review requires full engineering team — AI cannot validate cross-platform data persistence

### Contingency notes

- **Small:** Simple logic gaps or confusing navigation destinations often surface only during developer review
- **Medium:** Technical constraints discovered during API review require Do logic refactoring not initially scoped
- **Large:** Real-time data latency and regulated industry consent flows add mandatory logic states that extend the map
- **XL:** Cross-platform state persistence logic at global scale is highly complex and frequently surfaces unscoped requirements

## Service Blueprint

**Model:** C — Design & Synthesis
**PhoenixDX approach:** The Service Blueprint is the most powerful tool for identifying operational bottlenecks. In Large and XL projects, this is less about UX and more about Systems Thinking and Operational Efficiency. Always include Evidence at the top of the Blueprint — the tangible proof the user receives at each step. If the Evidence doesn't match the User Action, the service will feel broken regardless of the UI.

## User Flows

**Double Diamond phase:** Develop
**Model:** C — Design & Synthesis
**PhoenixDX approach:** User Flows are the Logic Map. Every click must have a destination and every Dead End must be eliminated before the design team starts building screens. For Large and XL projects, use a Numbered Node system — every screen and decision point gets a unique ID (e.g. 1.0, 1.1, 1.2). This allows Designers, Developers, and QA to refer to the exact same logic point without confusion.

| Large | 15+ flows, multi-platform, edge cases | 13d | 9d | 1.0x |
| Extra Large | Modular flow architecture, global enterprise | 30d | 21d | 1.2x |

### Phase breakdown — see Master Excel Model C - User flows tab for full phase details

### Deliverables

- **Medium:** Logic flow maps for 5 journeys with Roadmap mapping, stakeholder presentation, and developer API review notes
- **Large:** Comprehensive Experience Maps with Wireflow thumbnails, QA-ready logic, and high-stakes sign-off workshop
- **XL:** Modular flow architecture document, numbered node system, final logic freeze confirmation, and executive readout

### Dependencies

## Hypothesis

## Design System

**Double Diamond phase:** Deliver
**Model:** C — Design & Synthesis
**PhoenixDX approach:** The Design System is the Single Source of Truth. In Large and XL projects, the focus shifts from making components to DesignOps — ensuring the system can be maintained and scaled by multiple squads without diverging. Never build a component without an Empty State, a Loading State, and an Error State. In enterprise software, these states make up 40% of the user experience.

### Effort by size

| Size | Scope trigger | Effort | AI-accelerated | Friction |
|------|--------------|--------|---------------|---------|
| Small | Basic UI Kit, core Atoms, single platform | 5d | 3.5d | 1.1x |
| Medium | Foundational library, Auto-layout, Variants | 12d | 8d | 1.1x |
| Large | Multi-platform, Design Tokens, DesignOps governance | 28d | 19d | 1.2x |
| Extra Large | Enterprise System of Systems, multi-brand, full docs site | 65d | 44d | 1.3x |

### Phase breakdown

**Brief Review**
- Small: 0.5d — Audit existing styles (Colors, Type), identify ~10 core components
- Medium: 1d — Audit 3–5 products, identify shared patterns, align on tech stack (e.g. React)
- Large: 2d — Deep-dive audit of legacy debt, align multi-brand requirements, token strategy
- XL: 4d — Global audit across all business units, review regional brand variances and infra

**First Draft**
- Small: 3d — Build basic UI Kit in Figma (Buttons, Inputs, Typography)
- Medium: 7d — Develop foundational components with Auto-layout, basic documentation
- Large: 15d — Multi-platform component library, define Design Tokens, complex organisms
- XL: 35d — Enterprise-grade System of Systems, multi-brand themes, full documentation site

**Internal Review**
- Small: 0.5d — Visual QA for consistency and accessibility (contrast)
- Medium: 1d — Peer review for component flexibility, initial dev-readiness check
- Large: 3d — Formal Stress Test with Engineering, accessibility audit (WCAG), token validation
- XL: 7d — Global Design Chapter review, ensuring cross-product and cross-platform logic

**Iteration**
- Small: 0.5d — Fix minor spacing or naming inconsistencies
- Medium: 2d — Refactor components based on technical realities from developers
- Large: 6d — Significant refactoring of complex components (e.g. Data Tables/Date Pickers)
- XL: 14d — Large-scale style refactoring across multiple product repositories

**Sign-off**
- Small: 0.5d — Final walkthrough with the primary stakeholder, file handover
- Medium: 1d — Formal presentation to the Product team, launch of the V1 UI Kit
- Large: 2d — High-stakes workshop, securing budget for ongoing maintenance (DesignOps)
- XL: 5d — Executive Gallery Walk, final approval for global enterprise implementation

### Deliverables

- **Small:** Basic Figma UI Kit covering core Atoms with file handover and primary stakeholder walkthrough
- **Medium:** V1 component library with Auto-layout components, basic documentation, and formal Product team launch
- **Large:** Multi-platform component library with Design Tokens, accessibility audit report, DesignOps governance plan, and engineering launch
- **XL:** Enterprise Design System with full documentation site, multi-brand token structure, global contribution model, and board-level Gallery Walk

### Dependencies

- **Small:** Approved high-fidelity designs, basic brand guidelines, single designer availability
- **Medium:** Approved high-fidelity designs for a single product, engineering tech stack confirmed, 2–3 designer team
- **Large:** Approved multi-platform high-fidelity designs, full brand guidelines, engineering team availability for token implementation
- **XL:** Global brand guidelines across all business units, multi-squad engineering availability, global design chapter leadership

### Scope guard rules

- Design System requires High-Fidelity Designs to be approved before starting
- If Accessibility/Risk is selected as strategic goal → Design System is **mandatory**
- Multi-brand or white-label requirements must be identified at Brief Review — they triple the First Draft phase
